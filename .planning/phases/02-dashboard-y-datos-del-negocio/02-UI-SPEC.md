# Phase 2 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope:** Login, app shell/nav (owner + superadmin) sobre el modelo **Tenant (grupo) → Negocio(s) (sucursales/peluquerías)**, selector de negocio en el shell del owner, Profesionales (list + create/edit + horario semanal + servicios que realiza + precio custom) operando sobre el negocio seleccionado, Servicios (list + create/edit + orden) sobre el negocio seleccionado, Perfil del negocio (datos + WhatsApp visible + granularidad de grilla) del negocio seleccionado, Superadmin `/admin` (Tenants list + create/edit/desactivar con solo `nombre`; dentro de cada Tenant, su lista de Negocios con create/edit/desactivar + WhatsApp config no-secreta).

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, badge inner padding |
| sm | 8px | Compact spacing: table cell vertical padding, gap between stacked time-block rows in the horario editor, gap between form label and input |
| md | 16px | Default spacing: form field vertical rhythm, card inner padding, gap between sidebar nav items |
| lg | 24px | Section padding: page header to content, card-to-card gap in grid layouts |
| xl | 32px | Layout gaps: sidebar width padding, gap between major page sections (e.g. "Datos generales" vs "Horario" vs "Servicios" within a professional's edit page) |
| 2xl | 48px | Major section breaks: page top margin under the topbar |
| 3xl | 64px | Page-level spacing: centered login card top offset on desktop |

Exceptions: none. The 8-point scale applies uniformly, including the dense weekly-schedule editor and admin tables — no ad-hoc off-scale values.

| Accent (10%) | Light `#2563EB` / Dark `#3B82F6` (override `--primary`, `--ring`, `--sidebar-primary`) | **Reserved strictly for:** primary action buttons (Iniciar sesión, Crear profesional, Crear servicio, Guardar cambios, Crear peluquería), the active/selected sidebar nav item (background tint + icon + text), focus rings on inputs, text links ("Ver detalle"), the active tab underline in Tabs, and the "on" state of Switch toggles (activo/inactivo). Never used for body text, borders, disabled states, or decorative fills. |
| Destructive | Light `#DC2626` (`oklch(0.577 0.245 27.325)`, shadcn default) / Dark `#EF4444`-ish (`oklch(0.704 0.191 22.216)`, shadcn default) | Destructive button variant ("Desactivar"), error inline validation text, error toast/alert icon and text, the title icon in deactivation AlertDialogs. ## Layout & Navigation (Claude's Discretion — resolved)

- **Owner shell:** left sidebar (fixed, 240px, `--sidebar`/`--sidebar-primary` tokens) with three sections: **Profesionales**, **Servicios**, **Negocio** (perfil del negocio). Comfortable density — 16px vertical rhythm between nav items, not compact. Topbar shows a **selector de negocio** (dropdown, `Select`/`DropdownMenu` component) that fija el `negocio_id` activo sobre el que opera todo el CRUD de la sesión (Profesionales/Servicios/Negocio leen y escriben solo sobre ese negocio); si el tenant tiene un único negocio, el selector colapsa a una etiqueta fija con el `negocio.nombre` (sin dejar de existir el concepto — persiste igual en cookie/URL, ver 02-CONTEXT.md D-13) en lugar de un dropdown interactivo. El resto del topbar (user menu: theme toggle, "Cerrar sesión") no cambia.
- **Superadmin shell:** completely separate sidebar at `/admin`, con navegación **jerárquica Tenant → Negocio(s)**: la sección principal es **Grupos** (lista de Tenants); al entrar a un Tenant se navega a su lista de **Negocios** (peluquerías de ese grupo), con un breadcrumb/back-link al listado de Grupos. Visually distinguished from the owner shell only by the route/content, not by a different color scheme (same design tokens — this is one app, one aesthetic; D-03 gates by middleware/role, not by visual re-skinning).
- **Responsive:** desktop-first (admin tool used at a front desk, not primarily on mobile). Sidebar collapses to an icon-only rail below `768px`; full off-canvas Sheet below `640px`.

## Weekly Schedule Editor — PRO-02 (Claude's Discretion — resolved)

- One row per day (Lunes–Domingo, always all 7 rows visible, never hidden).
- Each row supports **multiple time blocks** (e.g. 09:00–13:00 and 16:00–20:00) rendered as a horizontal list of `[time input] – [time input] [remove ×]` chips with an `sm`(8px) vertical gap between stacked blocks within the same day.
- "+ Agregar bloque" (ghost button, 12px label) appends a block to that day.
- "Copiar a todos los días" button (secondary variant) copies the first configured day's blocks to the remaining 6 days, with an inline undo toast (Sonner) for 5 seconds after applying.
- Time inputs are native `<input type="time">` styled via shadcn `Input`, stepped in minutes matching the negocio's grid granularity (BIZ-03: 15 or 30) where the browser supports a `step` attribute; falls back to free entry otherwise.
- A day with zero blocks reads as "Cerrado" (12px, muted-foreground color, italic) instead of an empty row.

## Services + Custom Price Matrix — PRO-03/04 (Claude's Discretion — resolved)

- Rendered as a `Table` inside the professional's edit page, one row per active servicio: `[checkbox "Realiza"] | Servicio | Precio base (read-only, muted) | Precio personalizado (Input, disabled unless checkbox checked, placeholder = precio base formatted per Formatting Conventions)`.
- Unchecking "Realiza" clears any custom price for that service on save (no separate confirmation — this is a simple field toggle, not a destructive account-level action).

## CRUD Interaction Pattern (Claude's Discretion — resolved)

| Entity | Pattern | Rationale |
|--------|---------|-----------|
| Profesional | Full-page create/edit (`/profesionales/nuevo`, `/profesionales/[id]/editar`) | Needs room for datos generales + horario semanal (multi-block) + servicios matrix — too much content for a modal |
| Servicio | `Dialog` (modal) create/edit | Simple 4-field form (nombre, descripción, precio, duración) — modal keeps the list in context |
| Perfil del negocio | Single settings page, inline edit + "Guardar cambios" (no separate create state — the negocio row always exists post-signup); opera siempre sobre el **negocio seleccionado** en el selector del topbar | It's a settings screen, not a list of records |
| Superadmin Tenant (grupo) | `Dialog` (modal) con un único campo: **`nombre`** (sin datos operativos ni WhatsApp — eso vive en Negocio) | Alta/edición de un grupo es una acción mínima y transaccional; el Tenant ya no tiene más datos que gestionar en Fase 2 |
| Superadmin Negocio (peluquería, dentro de un Tenant) | `Dialog` (modal) with two sections in one form: "Datos generales" (nombre, dirección, teléfono, timezone, granularidad) and "WhatsApp" (phone_number_id, waba_id, número visible — no token field, per D-04) | Quick, transactional admin task; token entry deferred to Phase 7 keeps this form short enough for a modal |

## Soft-Delete Presentation (Claude's Discretion — resolved)

- No separate "archived" tab. Inactive rows stay in the same table, visually muted (reduced opacity ~60%, no accent color), with a neutral-gray "Inactivo" badge (NOT the destructive red — deactivation is a normal state, not an error).
- A segmented `Tabs` control at the top of each table: "Todos / Activos / Inactivos" filters the view.
- The activation toggle is a `Switch` directly in the row. Turning **off** (deactivating) opens the destructive `AlertDialog` from the Copywriting Contract. Turning **on** (reactivating) applies immediately, no confirmation.

## Service Ordering — SVC-02 (Claude's Discretion — resolved)

- Drag-and-drop reordering directly in the Servicios table via a drag-handle icon (`GripVertical` from lucide-react) at the left of each row, using `@dnd-kit/sortable`. Drop persists the new `orden` value immediately (optimistic UI + toast on failure to roll back).

## Visual Hierarchy & Accessibility (Dimension 2 — resolved)

- **Focal point per primary screen:**
  - **Login:** the centered login `Card` is the sole focal point; within it, the "Iniciar sesión" primary CTA (accent) draws the eye after the two fields.
  - **Landing after login (Profesionales list — the owner's default route):** the eye lands first on the **"+ Nuevo profesional"** primary CTA (only accent-colored element in the top-right of the page header), then flows to the table below. On an empty state, the empty-state heading + its inline CTA is the focal point.
  - **Superadmin `/admin` (Grupos/Tenants list, ruta raíz de `/admin`):** same pattern — "+ Nuevo grupo" primary CTA is the single focal accent, then the tenants table. **Dentro de un grupo (lista de Negocios de ese Tenant):** "+ Nueva peluquería" primary CTA is the single focal accent, then the negocios table.
  - Every page has exactly **one** primary (accent) action; secondary actions use neutral/ghost variants so the accent never competes with itself.
- **Icon-only control accessibility (MANDATORY for the executor):** every icon-only control carries an `aria-label` (or visually-hidden `sr-only` text) matching its action, and a visible focus ring (accent `--ring`). Specifically:
  - Theme toggle (Sun/Moon) → `aria-label="Cambiar tema"`.
  - Drag handle (`GripVertical`) → `aria-label="Reordenar servicio"` and keyboard-operable via `@dnd-kit` sortable keyboard sensor.
  - Time-block remove (`×`) → `aria-label="Quitar bloque horario"`.
  - Any future icon-only button follows the same rule — no unlabeled icon buttons ship.
