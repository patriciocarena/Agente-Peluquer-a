Vault (SEC-01, decisión LOCKED D-01/D-02), y aplicarla en vivo contra bdgufnitakelyialjoqg

tokens son NULL, ver Runtime State Inventory de 07-RESEARCH.md).

tiene contra qué correr, y la verificación de fase daría un falso verde (los tipos vienen
de config generada, no de la DB live — ver [BLOCKING] schema push).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| app-code (service_role) → schema `vault` | El único acceso permitido es vía los wrappers `public` SECURITY DEFINER; `vault` nunca se expone por PostgREST |
| DB dump / réplica / backup → atacante | Superficie por la que un token en claro se filtraría si viviera en una columna plana |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-07-01 | Information Disclosure | `negocio.whatsapp_token` (columna plana) | high | mitigate | Dropear la columna plana; guardar solo `whatsapp_token_secret_id`; el secreto vive authenticated-encrypted en `vault.secrets` |
| T-07-02 | Elevation of Privilege | wrappers `public.*_whatsapp_token*` | high | mitigate | SECURITY DEFINER + `SET search_path=''` + `REVOKE ALL FROM PUBLIC` + `GRANT EXECUTE TO service_role` (nunca authenticated) — idéntico a auth_negocio_ids() |
| T-07-05 | Information Disclosure | statement logging de Postgres al llamar vault.create_secret | medium | accept | No hay tokens reales hoy; chequeo operativo diferido a antes del primer token real (Assumption A3). No bloquea esta fase |
| T-07-SC | Tampering | npm/pip/cargo installs | low | accept | Esta fase no instala paquetes nuevos (Vault es extensión Postgres vía SQL); no hay superficie de supply-chain |
