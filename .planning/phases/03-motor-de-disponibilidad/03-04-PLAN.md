## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-08 | Tampering | computeSlots / ventana de reserva | mitigate | Los límites 60min/30d se enforzan DENTRO de computeSlots (paso 4, desde constants.ts), no como sugerencia de UI — defensa contra input manipulado (p.ej. prompt-injection del bot en Fase 6, BOT-11) que pida una fecha fuera de ventana. Test explícito en ambos bordes. |
| T-03-09 | Information Disclosure | computeSlots / scoping por negocio | mitigate | El input requiere `negocioId`; documentar que el caller debe pasar SOLO filas de ese negocio. Opcional (recomendado): aserción interna que falle ruidosamente si alguna fila de `data` tiene un `negocio_id` distinto al `input.negocioId`, para no computar disponibilidad cross-negocio silenciosamente si un caller tiene un bug. |
| T-03-10 | Tampering (integridad) | filtro busy (Pitfall 4) | mitigate | El filtro `estado IN ('pendiente','confirmado')` debe ser el inverso lógico exacto del `WHERE estado != 'cancelado'` del constraint DB; si diverge, el motor ofrece slots que el DB rechaza. Test cancelado-libera / pendiente-bloquea cubre esto. |
