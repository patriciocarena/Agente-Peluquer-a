<objective>
Extender el motor compartido `@turnosbot/availability-engine` con las dos capacidades que la fase 4 necesita del backend de dominio, SIN crear una segunda implementación de disponibilidad (AVAIL-04):

1. **D-08 — bypass de ventana de reserva:** un flag opcional `skipBookingWindow?: boolean` (default `false`, preserva el comportamiento del bot) que, cuando es `true`, hace que `computeSlots` NO aplique el filtro `BOOKING_MIN_LEAD_MINUTES`/`BOOKING_MAX_ADVANCE_DAYS`. Habilita que el dueño cargue turnos "para ahora mismo" o a más de 30 días (D-07). Backing de APPT-06.
2. **D-14 — `rescheduleAppointment`:** función nueva en `booking.ts`, hermana de `bookAppointment`, que revalida el nuevo horario contra `computeSlots(freshData)` **excluyendo el propio turno** de los turnos activos que bloquean, y hace `UPDATE turno SET inicio, fin, profesional_id WHERE id = turnoId` (nunca cancela+crea). Backing de APPT-05. Esta misma función la reusará el bot en Fase 6 (BOT-10).

Purpose: El dashboard (Planes 03-07) y el bot (Fase 6) deben compartir exactamente un motor. Estas extensiones son aditivas y de default-preservador: el bot que hoy no pasa `skipBookingWindow` sigue con la ventana intacta.
Output: `types.ts` extendido, `computeSlots.ts` con branch condicional, `rescheduleAppointment` + `rescheduleAppointmentInputSchema` en `booking.ts`, barrel `index.ts` actualizado, tests unitarios verdes, y `scripts/verify-reschedule.ts` (escrito, ejecución live gated como en Fase 3).
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller (dashboard action / bot tool) → engine | El caller pasa `negocioId` y filas ya-fetcheadas; el motor es puro y NO puede enforcar aislamiento por sí mismo (T-03-01) |
| engine → Postgres (via deps.supabase inyectado) | La GiST EXCLUDE es la última línea anti-doble-reserva; el UPDATE de reschedule cruza este borde |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-01 | Tampering | rescheduleAppointment UPDATE vs GiST EXCLUDE (race de reagendado concurrente) | high | mitigate | Revalidación de freshness contra `computeSlots(dataExcludingSelf)` pre-UPDATE + traducción de `23P01`→`slot_taken` vía `isSlotTakenConcurrently` (mismo patrón probado de bookAppointment); `scripts/verify-reschedule.ts` verifica live que el UPDATE dispara 23P01 (A1) |
| T-04-02 | Elevation of Privilege | Bypass accidental de la ventana en el path del bot (skipBookingWindow default permisivo) | high | mitigate | Campo opcional con default falsy en cada capa; test de regression (Pitfall 1) prueba que omitir el campo = comportamiento del bot intacto; solo el dashboard pasa `true` explícito |
| T-04-03 | Information Disclosure | Filas cross-negocio en `dataExcludingSelf` | medium | mitigate | `assertScopedToNegocio` (ya existente) corre dentro de `computeSlots` sobre `dataExcludingSelf`; el spread-and-filter preserva el scoping del `freshData` original |
| T-04-04 | Tampering | Input LLM-driven manipulable (turnoId/serviceIds basura) en Fase 6 | medium | mitigate | `rescheduleAppointmentInputSchema` con `uuidLike` corta input malformado en el borde del paquete (V5), igual que bookAppointment |
