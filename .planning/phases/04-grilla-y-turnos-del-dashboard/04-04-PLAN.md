<objective>
Las Server Actions de turno que SÍ usan el motor compartido (dependen de Plan 01 y del helper de fetch de Plan 03). Único camino de escritura = `bookAppointment`/`rescheduleAppointment` (D-11/D-14), nunca un insert/update paralelo de disponibilidad.

1. `app/actions/turnos.ts`:
   - `crearTurnoManual` (APPT-06, D-10/D-11): llama `bookAppointment({ ...input, negocioId, skipBookingWindow: true }, { supabase, freshData })`. El bypass de ventana (D-07) permite "ahora mismo"/fechas lejanas. Luego, si el turno debe quedar `confirmado` (alta manual del dueño), actualiza el estado a `confirmado` (bookAppointment inserta en `pendiente`).
   - `cancelarTurno` (APPT-04, D-06/D-12): `UPDATE turno SET estado='cancelado' WHERE id=... AND negocio_id=...`. No borra la fila (historial). `revalidatePath` para que la celda vuelva a libre al instante.
   - `reagendarTurno` (APPT-05, D-14): trae los `serviceIds` originales del turno (turno_servicio), y llama `rescheduleAppointment({ negocioId, turnoId, profesionalId, serviceIds, inicio, fin }, { supabase, freshData })`.
2. `app/actions/slots.ts`:
   - `obtenerSlotsDisponibles({ serviceIds, fecha, profesionalId? })` (D-10/D-13): wrapper de `computeSlots` con `skipBookingWindow: true`, sobre `buildAvailabilityData`. Devuelve `AvailableSlot[]`.
   - `profesionalesElegibles(serviceIds)` (Pitfall 6 / Open Question 3): lista los profesionales con fila en `profesional_servicio` para TODOS los servicios pedidos — así el selector de profesional nunca ofrece a alguien que no hace el servicio.

Purpose: Concentrar la lógica de escritura/consulta de disponibilidad del dashboard en Server Actions que delegan 100% al motor, cerrando el gap de elegibilidad profesional×servicio antes de construir la UI.
Output: dos módulos de Server Actions.
</objective>
