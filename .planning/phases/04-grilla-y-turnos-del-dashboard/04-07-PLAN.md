## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| navegador → Server Component (searchParams) | `?fecha=` es input no confiable; se valida como YYYY-MM-DD y no se usa para scoping (el negocio sale de getNegocioActivo) |
| Server Component → engine/DB | El fetch se scopea por negocio_id (Plan 03); computeSlots corre con filas ya-scopeadas |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-24 | Elevation of Privilege / Information Disclosure | `negocio_id` derivado de input en la grilla | high | mitigate | `page.tsx` deriva el negocio SOLO de `getNegocioActivo()`; `?fecha=` nunca influye en el scoping; RLS + buildAvailabilityData scopean el fetch |
| T-04-25 | Tampering | `?fecha=` malformado (inyección/valor inválido) | medium | mitigate | Validar `fecha` como `YYYY-MM-DD` antes de usarla; fallback a hoy en TZ del negocio; nunca interpolar cruda en queries (se pasa a computeSlots como string de fecha, no a SQL) |
| T-04-26 | Information Disclosure | Render de datos de otro negocio en la grilla | high | mitigate | `assertScopedToNegocio` del motor lanza ante fila cruzada; el fetch ya viene scopeado por RLS + `.eq("negocio_id", ...)` |
| T-04-27 | Information Disclosure | Free-text (cliente, motivo) en celdas | low | accept | Escapado por defecto de React + `title` nativo; sin HTML crudo |
