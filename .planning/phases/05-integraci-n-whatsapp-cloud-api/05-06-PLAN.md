<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Meta (internet) → POST /webhooks/whatsapp | Public endpoint; any actor can POST — only a valid HMAC proves Meta origin |
| raw bytes → JSON parse | Signature must be verified over the exact received bytes before any parse |
| GET handshake → subscription | verify_token must match the env-configured secret |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-01 | Spoofing | POST /webhooks/whatsapp | high | mitigate | Verify `X-Hub-Signature-256` over `rawBody` (captured by the buffer content-type parser registered before routing) with `verifyWhatsappSignature`; invalid/missing → 403, no enqueue, no parse (WA-01) |
| T-05-03 | Tampering | Meta retry duplication | high | mitigate | Enqueue with `singletonKey = messages[0].id` (first dedup layer) and return 200 immediately; the durable DB constraint (plan 05-01/05-05) is the backstop (WA-03) |
| T-05-04 | Denial of Service | public webhook flood/replay | medium | mitigate | `@fastify/rate-limit` on the app + `@fastify/helmet` security headers (D-12); enqueue-then-200 keeps the handler fast so a slow downstream cannot exhaust connections |
| T-05-14 | Spoofing | GET subscription handshake | medium | mitigate | `hub.verify_token` compared against `env.WHATSAPP_VERIFY_TOKEN`; mismatch → 403 (D-05) |
</threat_model>
