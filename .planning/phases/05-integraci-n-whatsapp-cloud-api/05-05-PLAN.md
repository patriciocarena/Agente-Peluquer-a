  <behavior>
    - Unknown phone_number_id → returns early, logs a discard, writes no cliente/conversacion/mensaje rows (D-07).
    - Non-message event (no messages[0]) → returns early, no writes.
    - Happy path → inserts one inbound mensaje (direccion 'entrante'), calls responder, and within the window inserts one outbound mensaje (direccion 'saliente') and calls sendWhatsappMessage exactly once.
    - Duplicate wa_message_id (insert returns error code 23505) → returns before responder/send; no second persist, no second send.
    - Window closed (now >= ventana_expira_at) → logs, does NOT call sendWhatsappMessage, does NOT insert an outbound mensaje.
  </behavior>
  <action>
    Create `apps/bot/src/queue/inboundWorker.ts` exporting `async processInboundWhatsappEvent(event: WhatsappWebhookEvent, deps?): Promise<void>`. Extract `phone_number_id` and the first message via the `payload.ts` helpers; return early if either is missing. Resolve the negocio strictly: `supabaseAdmin.from("negocio").select("id, timezone").eq("whatsapp_phone_number_id", phoneNumberId).maybeSingle()` — on no match, log a warning and return, writing nothing (Pattern 3, D-07: never guess a tenant, never default). Then `findOrCreateCliente(negocio.id, message.from)`, `findOrCreateConversacion(negocio.id, clienteId)`. Insert the inbound message through the write accessor `negocioScoped(negocio.id).insertMensaje({ conversacion_id, direccion: "entrante", wa_message_id: message.id, contenido: message })` — use the Spanish value 'entrante' (the CHECK constraint rejects other values; 05-PATTERNS.md correction). If the insert returns Postgres error code 23505 (unique_violation on wa_message_id), log "already processed" and return BEFORE calling responder or sending (this is the durable dedup backstop, WA-03) — outbound send is conditioned on this insert succeeding, so a duplicate short-circuits every side effect. Call `reply = await responder(conversacion, message.text?.body ?? "")`. Gate the outbound: if `Date.now() < new Date(conversacion.ventana_expira_at).getTime()`, `await sendWhatsappMessage(negocio.id, message.from, reply)` then insert the outbound `negocioScoped(negocio.id).insertMensaje({ conversacion_id, direccion: "saliente", contenido: { text: { body: reply } } })`; else log the closed-window skip (REMIND-01 out of scope, D-09). To make this unit-testable without a live DB/queue, accept an optional injected `deps` param defaulting to the real collaborators (mirror booking.ts's `BookAppointmentDeps` optional-deps pattern): `{ supabaseAdmin, findOrCreateCliente, findOrCreateConversacion, responder, sendWhatsappMessage, negocioScoped, log }`. Lead with a doc-comment citing WA-02/03/04/05, D-07/09/11, Pattern 3/4/5. Create `apps/bot/src/queue/inboundWorker.test.ts` covering all five behaviors using injected fake deps (a fake supabaseAdmin/negocioScoped returning canned rows, a spy sendWhatsappMessage) — no network, no DB.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| enqueued payload → tenant selection | The payload's phone_number_id must select a tenant only via a DB lookup, never be trusted directly |
| worker → Postgres (service_role) | RLS bypassed; isolation is application-enforced |
| pg-boss → Postgres connection | Must be session-mode (5432); transaction pooling (6543) silently degrades dedup/locks |

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |

| T-05-02 | Elevation of Privilege / Info Disclosure | tenant resolution by phone_number_id | high | mitigate | Resolve negocio strictly via `.eq("whatsapp_phone_number_id", ...)`; unmatched id → log + discard with zero writes, never a default/fallback tenant (D-07, Pitfall 3) |
| T-05-03 | Tampering (data integrity) | duplicate processing under Meta 7-day retry | high | mitigate | Durable dedup: a 23505 unique-violation on `wa_message_id` short-circuits before responder/send; outbound is conditioned on the inbound insert succeeding (Pattern 4, WA-03) |
| T-05-13 | Denial of Service / integrity | pg-boss on wrong connection mode | high | mitigate | boss.ts hard-fails without `SUPABASE_DB_URL` and documents the port-5432 session-mode requirement; 6543 transaction pooling is explicitly forbidden |
