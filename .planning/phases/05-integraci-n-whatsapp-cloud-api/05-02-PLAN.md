<objective>
Build the two pure, dependency-light building blocks of WA-01: constant-time `X-Hub-Signature-256` verification over the raw request body, and a zod schema that validates the parsed Meta webhook payload shape. Both are pure functions with no DB/queue/HTTP coupling, so they are fully unit-testable and parallelizable.

Purpose: the webhook route (plan 05-06) and the worker (plan 05-05) depend on these; getting the signature length-guard and the payload shape right here removes the single most common WhatsApp webhook bug (re-serialization) and a DoS crash (length-mismatch throw).
Output: `signature.ts` (+ test), `payload.ts` (+ test).
</objective>

  <behavior>
    - Correctly signed body (HMAC-SHA256 of the exact bytes with the app secret, `sha256=` prefix) → returns true.
    - Same signature, one byte of the body changed → returns false.
    - Correct body, wrong secret → returns false.
    - Header undefined → returns false (no throw).
    - Header without the `sha256=` prefix → returns false.
    - Header present but shorter/longer than the expected hex digest → returns false, never a RangeError (Pitfall 2).
  </behavior>
  <action>
    Create `apps/bot/src/whatsapp/signature.ts` exporting `verifyWhatsappSignature(rawBody: Buffer, signatureHeader: string | undefined, appSecret: string): boolean` per 05-RESEARCH.md Pattern 2: guard `signatureHeader?.startsWith("sha256=")` first; compute `createHmac("sha256", appSecret).update(rawBody).digest("hex")` from `node:crypto`; build Buffers for expected and received hex; compare buffer lengths and return false on mismatch BEFORE calling `timingSafeEqual` (the length guard prevents the RangeError DoS in Pitfall 2); return `timingSafeEqual(expectedBuf, receivedBuf)`. The input MUST be the raw Buffer bytes — the header comment must state that verifying over re-serialized JSON silently breaks (Pitfall 1) and that this is why the caller passes rawBody, never `JSON.stringify(request.body)`. Lead with a doc-comment citing WA-01, D-06, Pitfall 1 and Pitfall 2, matching booking.ts's rationale-first style. Create `apps/bot/src/whatsapp/signature.test.ts` covering all six behaviors above, importing `./signature.js`, computing a reference HMAC in-test with a known secret and body. No mocking needed (pure function).
  </action>
